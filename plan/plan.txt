## 2. Multi-Tenant Business Architecture
AGS acts as a "Landlord" - we provide the software, organizations run their sports.

    • Tier 1: Local/Grassroots (Transactional)
        ◦ Users: Schools, local clubs, neighborhood organizers
        ◦ Revenue: Commission on registration fees + platform fees
        ◦ Self-Service: Complete autonomy for organizers

    • Tier 2: Professional/Enterprise
        ◦ Users: Federations, pro leagues, state sports boards
        ◦ Revenue: Annual license fee or higher commission tiers
        ◦ Dedicated instances with custom branding

## 4. Technical Architecture (Django + Firebase)

### Core Stack
    • Backend: Django + Django REST Framework
    • Authentication: Firebase Authentication for MVP login
    • Database: PostgreSQL with org_id-based tenant filtering
    • Frontend: React + Vite + Tailwind CSS
    • Hosting: Django backend on standard cloud platform, React frontend on Vercel/Netlify or Firebase Hosting
    • Storage: Firebase Storage for uploads if needed
    • Realtime options: Django Channels/Redis for chat or Firebase Realtime DB for lightweight state

### Multi-Tenancy Implementation
    • Single PostgreSQL database with org_id tenant field
    • Django middleware and DRF permissions enforce tenant isolation
    • Optionally add PostgreSQL RLS later for stronger data separation
    • Authorization rule example:
      - Users can only access resources where resource.org_id == user.org_id
      - Super-admin role can access all tenants

### Authentication Strategy
    • Use Firebase Auth for user sign in, registration, email verification
    • Frontend uses Firebase SDK to sign in and obtain ID tokens
    • Django verifies Firebase ID tokens on incoming requests
    • Django maps Firebase UID/email to local user records and issues app sessions or API tokens
    • Keep core business data in Django, not Firebase

### Realtime Architecture
    • Use simple polling or Django Channels for MVP live updates
    • Use Firebase Realtime DB / Firestore only for secondary features like chat or spectator status if needed
    • Keep match/multitenant data in PostgreSQL and update UI via WebSocket or frontend polling

## 7. Strategic Business Insights

### The "Mafia Offer" - Our 10x Feature
Solve age-fraud in youth sports with instant digital player verification. If we can verify a player's age and identity in one click, leagues will adopt us just for that.

### The Data Moat
Verified player and match data becomes the source of truth for scouts, leagues, and referees. That creates network effect and defensibility.

### Land and Expand
Start with tournament brackets and live match scoring as the hook. Once data and workflows are locked in, expand into chat, analytics, and player verification.

### Self-Serve Infrastructure
Tenant onboarding must be zero-touch. Organizers should start a tournament without support intervention.

### The "Noisy Neighbor" Problem
Protect small tenants by limiting requests per org and optimizing database queries.

### Data Residency (Future)
Design the app so tenant data can be hosted regionally later if needed.
